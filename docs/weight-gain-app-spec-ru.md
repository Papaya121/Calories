# Документация: Calories (Weight Gain Tracker)

## 1. Цель продукта
Приложение помогает системно набирать вес через учёт питания по фото:
- пользователь фотографирует еду и при необходимости добавляет комментарий;
- backend отправляет фото+комментарий в OpenAI API;
- приложение получает примерные КБЖУ, название и описание блюда;
- запись сохраняется в дневник;
- в календаре и дневном экране показывается, что и сколько было съедено, плюс суммарные калории.

## 2. Формат продукта
- Клиент: PWA (мобильный web-first, стиль максимально близкий к iOS).
- Backend: NestJS.
- База данных: PostgreSQL.
- Хранение изображений: S3-совместимое хранилище (S3/Cloudflare R2/MinIO).

## 3. Пользовательский сценарий (MVP)
1. Первый запуск:
- пользователь задаёт 6-значный passcode;
- пользователь может включить биометрию (Face ID/Touch ID через WebAuthn);
- создаётся сессия и сохраняется refresh-токен.

2. Добавление еды:
- пользователь открывает экран добавления;
- делает фото (или выбирает из галереи) + при желании пишет комментарий;
- backend отправляет данные в OpenAI API с требованием вернуть структурированный JSON;
- пользователь видит предпросмотр результата (название, описание, КБЖУ), редактирует при необходимости;
- запись сохраняется.

3. Просмотр прогресса:
- дневной экран: список приёмов пищи + итог КБЖУ за день;
- календарь: краткие итоги по каждому дню (калории и число записей).

4. Повторный вход:
- локальная разблокировка через passcode или биометрию;
- backend обновляет access-token через refresh-token;
- пользователь продолжает работу без логина/пароля.

## 4. Функциональные требования

### 4.1 Приёмы пищи
- Загрузка 1+ фото в запись (MVP можно начать с 1 фото).
- Опциональный комментарий к фото.
- AI-анализ с полями:
  - `dish_name`
  - `dish_description`
  - `calories_kcal`
  - `protein_g`
  - `fat_g`
  - `carbs_g`
  - `confidence` (0..1)
- Ручное редактирование результата перед сохранением.
- CRUD для записей о еде.

### 4.2 Календарь и статистика
- Просмотр месячного календаря.
- Индикатор, что в день есть записи.
- Открытие дня по нажатию:
  - список приёмов;
  - сумма КБЖУ за день.
- Диапазон дат для просмотра истории.

### 4.3 Аутентификация и сессии
- Без логина/пароля в MVP UX.
- Разблокировка:
  - WebAuthn (platform authenticator: Face ID/Touch ID, где доступно);
  - fallback: локальный passcode.
- Backend-token модель:
  - короткоживущий access JWT (например, 15 мин);
  - refresh-token в HttpOnly cookie (например, 30 дней, rotation).
- Сейчас 1 аккаунт, но модель должна поддерживать multi-user.

## 5. Нефункциональные требования
- UI должен выглядеть “айфоновским” (плавно, аккуратно, без грубых анимаций).
- Время ответа AI-анализа: целевой p95 до 8-12 секунд (зависит от модели и размера фото).
- Безопасное хранение токенов и фото.
- Возможность горизонтального масштабирования backend.
- Наблюдаемость: structured logging + error tracking.

## 6. Предложенный стек

## 6.1 Frontend (выбор)
Рекомендуется: `Next.js (App Router) + TypeScript + Tailwind CSS + React Query + Zustand`.

Почему:
- удобная разработка PWA и API-интеграций;
- простая адаптация под будущую web+mobile стратегию;
- хорошая экосистема для production.

PWA:
- `manifest.webmanifest`;
- Service Worker (Workbox/next-pwa);
- offline-cache для статики и последних просмотренных дней;
- add-to-home-screen UX.

## 6.2 Backend
`NestJS + Prisma + PostgreSQL + Redis (опционально для очередей/кэша)`.

Интеграции:
- OpenAI API для vision-анализа;
- S3 storage для фото.

## 7. Архитектура (высокий уровень)
1. Клиент отправляет фото+комментарий в backend.
2. Backend сохраняет фото в storage.
3. Backend вызывает OpenAI API с image input и строгим JSON-форматом ответа.
4. Backend валидирует ответ (zod/class-validator), нормализует КБЖУ.
5. Backend возвращает результат на клиент для подтверждения.
6. После подтверждения запись и нутриенты сохраняются в БД.
7. Клиент получает дневные/календарные агрегаты через API.

## 8. Контракты API (MVP)

Базовый префикс: `/api/v1`

### 8.1 Auth
- `POST /auth/bootstrap`  
  Первый запуск: создать passcode hash, (опционально) зарегистрировать WebAuthn credential.
- `POST /auth/passcode/login`  
  Вход по passcode.
- `POST /auth/webauthn/options`  
  Получить challenge.
- `POST /auth/webauthn/verify`  
  Подтвердить assertion и создать сессию.
- `POST /auth/refresh`  
  Выдать новый access token.
- `POST /auth/logout`  
  Инвалидировать refresh token.

### 8.2 Meals
- `POST /meals/analyze`  
  Body: `multipart/form-data` (photo, comment?)  
  Response: AI-предсказание КБЖУ и описания.
- `POST /meals`  
  Сохранить подтверждённую запись.
- `PATCH /meals/:id`  
  Редактировать запись.
- `DELETE /meals/:id`  
  Удалить запись.
- `GET /meals/:id`  
  Получить запись.

### 8.3 Calendar / Day
- `GET /calendar?from=YYYY-MM-DD&to=YYYY-MM-DD`  
  Дневные итоги по диапазону.
- `GET /days/:date`  
  Детали за день (список + суммы).

### 9.2 Индексы

- `sessions(user_id, expires_at)`
- `auth_webauthn_credentials(user_id, credential_id)`

## 10. Интеграция с OpenAI API

### 10.1 Принципы
- Использовать vision-модель через Responses API.
- Требовать строго структурированный JSON-ответ.
- Добавить server-side валидацию и fallback:
  - если данных недостаточно, возвращать `needs_user_confirmation = true`;
  - всегда позволять ручную правку перед сохранением.

### 10.2 Пример целевого JSON
```json
{
  "dish_name": "Chicken rice bowl",
  "dish_description": "Rice bowl with grilled chicken and vegetables",
  "calories_kcal": 620,
  "protein_g": 38,
  "fat_g": 18,
  "carbs_g": 72,
  "confidence": 0.74
}
```

### 10.3 Ограничения точности
- КБЖУ по фото всегда приблизительные.
- В UI явно показывать “оценка, не медицинское измерение”.

## 11. UX/UI и визуальный стиль (iPhone-like)

### 11.1 Визуальные принципы
- Чистая композиция, акцент на контент (фото и суточные итоги).
- Мягкие скругления, тонкие тени, полупрозрачные слои.
- Минимум визуального шума и декоративных элементов.

### 11.2 Типографика
- Основной шрифт: `SF Pro Display/Text` (с fallback на system sans).
- Иерархия: крупные числа калорий, компактные подписи макросов.

### 11.3 Цвета
- Нейтральная светлая база.
- 1 акцентный цвет (например, iOS blue) для CTA и активных состояний.
- Макросы кодировать мягкими, но различимыми цветами.

### 11.4 Анимации
- Только короткие и функциональные:
  - 160-240ms для переходов;
  - spring-анимация для карточек/модалок;
  - skeleton/shimmer для AI-анализа.
- Запрещены “топорные” анимации: резкие скейлы, длинные easing, дёрганые переходы.

### 11.5 Основные экраны
- Экран логина/разблокировки (passcode + Face ID кнопка).
- Главный экран “Сегодня” (итоги + список приёмов).
- Экран добавления еды (камера/галерея, комментарий, предпросмотр AI).
- Календарь (месяц + переход в день).
- Экран записи (фото, описание, КБЖУ, редактирование).

## 12. Безопасность
- Passcode хранить только в виде устойчивого hash (Argon2id/scrypt).
- Refresh-token хранить в HttpOnly secure cookie.
- Access-token короткоживущий.
- Signed URL для доступа к фото.
- Rate-limit на auth и upload endpoints.

## 13. Наблюдаемость и качество
- Логи: request-id, user-id, latency, AI model name, token usage.
- Ошибки: централизованный error tracking.
- Метрики:
  - число анализов/день;
  - median/p95 времени анализа;
  - доля ручных исправлений после AI;
  - retention по дням.

## 14. План реализации

### Фаза 1 (MVP)
- Auth (passcode + refresh/access).
- Загрузка фото и AI-анализ.
- CRUD записей и дневные итоги.
- Календарь с суммой ккал за день.
- PWA-инсталляция и базовый offline.

### Фаза 2
- WebAuthn (Face ID/Touch ID) как основной способ разблокировки.
- Улучшение AI-промптов и повторный анализ.
- Недельные/месячные отчёты.

### Фаза 3
- Полноценный multi-user.
- Цели по весу, уведомления, рекомендации по питанию.

## 15. Критерии готовности MVP
- Пользователь может за <20 секунд добавить запись еды с фото и получить КБЖУ.
- Дневной экран корректно считает сумму калорий и макросов.
- Календарь показывает заполненные дни и корректные итоги.
- Повторный вход работает через passcode, сессия восстанавливается по refresh-токену.
- UI стабильно работает как PWA на iPhone (Safari + Add to Home Screen).
